## Option 1: Generate a Web Link

The simplest way to share: generate an access link with one click.

**Steps:**

1. Open your application (e.g., "TechStore AI Customer Service")
2. Click "Publish" in the top-right corner
3. After confirming the publish, click the "Access" button next to "Run"
4. You'll get a link: `https://udify.app/chat/xxxxxxxx`

Share this link with your colleagues and they can start using it immediately.

*"Try this link. Let me know if there are any issues."*

---

**Customization options:**

In the "Publish" settings, you can adjust:
- Application name and icon
- Welcome message
- Input placeholder text
- Access password (optional)

Lynn added a password to the link and only shared it with colleagues for internal testing.

## Option 2: Embed in a Website

The CS team lead came to Lynn: *"Can we put this AI customer service on our website? So customers can ask questions directly."*

Dify supports embedding applications into any webpage.

**Steps:**

1. In the application settings, find "Embed in Website"
2. Copy the generated embed code:

```html
<script>
  window.difyChatbotConfig = {
    token: 'your-token-here'
  }
</script>
<script src="https://udify.app/embed.min.js" defer></script>
```

3. Add this code to your website's HTML

The above is the minimal version. In practice, use the code generated by the "Embed in Website" page in Dify's dashboard — it will automatically fill in your token and the correct script URL.

**Result**: A chat bubble appears in the bottom-right corner of the website. Customers click it and start chatting with the AI.

## Option 3: API Integration

The marketing team had a more complex need: *"We want to integrate the content factory into our internal system. Fill in product info in the system → click Generate → it calls your tool automatically → results are populated back."*

This requires an API.

**What is an API?**

Simply put, an API is "how machines talk to each other."

- When you use Dify through the web interface, that's a human talking to a machine
- When your system calls Dify through an API, that's a machine talking to a machine

**Getting the API:**

1. In the application settings, find "API Access"
2. Create an API Key (this is your credential — keep it safe)
3. View the API documentation

**API call example:**

```bash
POST https://api.dify.ai/v1/workflows/run
Headers:
  Authorization: Bearer your-api-key
  Content-Type: application/json

Body:
{
  "inputs": {
    "product_info": "TechBuds Pro 3 True Wireless ANC Earbuds...",
    "platforms": ["Blog", "Instagram", "Twitter/X"]
  },
  "user": "marketing_system"
}
```

**Response:**
```json
{
  "data": {
    "outputs": {
      "content": "[Blog]...[Instagram]...[Twitter/X]..."
    }
  }
}
```

## Option 4: DSL Export

Mike from Finance came to Lynn: *"That weekly report generator of yours is pretty nice. Can you make me a finance version?"*

Lynn thought about it: *"I can give you the framework — just tweak it a bit and you're good to go."*

**What is DSL?**

DSL (Domain Specific Language) is the format Dify uses to save application configurations. It saves your entire Workflow — all nodes, connections, and settings — into a single file.

**Exporting DSL:**
1. In the application list, click the "..." menu on the application
2. Select "Export DSL"
3. Download a `.yml` file

**Importing DSL:**
1. On the application list page, click "Create Application"
2. Select "Import DSL File"
3. Upload the `.yml` file

---

Lynn sent the weekly report generator's DSL to Mike with a note:

*"Don't touch the node structure — just change 'operations report' to 'finance report' in the prompts and adjust the output format."*

Mike spent two hours tinkering and produced a "Finance Weekly Report Generator."

**Benefits of DSL:**
- Share application frameworks without starting from scratch
- Version management — roll back if something breaks
- Team collaboration — everyone can iterate on the same version

::: tip Lynn's Habit
Before making major changes to a Workflow, always export a DSL backup first.
:::

## Option 5: Publish as an MCP Server

A developer colleague in IT came to Lynn: *"I'm coding in Cursor and I often need to look up TechStore product info. Can you let me query your knowledge base directly from Cursor?"*

Lynn wasn't sure how to do it at first, but then discovered that Dify supports **publishing applications as an MCP Server**.

**What is MCP?**

MCP (Model Context Protocol) is a standard protocol that enables AI tools to call each other. Think of it as "the USB port for AI tools" — as long as both sides support MCP, they can connect.

Tools that currently support MCP include Claude Desktop, Cursor, Windsurf, and more — and the list is growing fast.

**How to publish:**

1. Open your application and go to Settings
2. Find the "MCP Server" option and enable it
3. Dify generates an MCP Server URL (including authentication info)
4. Configure this URL in Claude Desktop or Cursor

Once configured, the developer colleague could query TechStore's product knowledge base directly from Cursor — no need to switch to the Dify interface or write API integration code.

::: tip Choosing Among the Five Publishing Options
- **Web link**: Simplest — great for non-technical users
- **Embed in website**: For end users — embed in your official site or product pages
- **API**: For developers — integrate into internal systems
- **DSL export**: Share application frameworks — team collaboration
- **MCP Server**: Connect to the AI tool ecosystem (Claude, Cursor, etc.)
:::

## Security & Access Control

After the applications were published, the IT security specialist came to Lynn: *"How do you control permissions for these AI tools?"*

Lynn organized what she knew:

**API Key security:**
- Don't put keys in frontend code
- Route API calls through a backend server
- Rotate keys regularly

**Access control:**
- Set access passwords for internal tools
- Track usage via the API's `user` parameter
- Restrict sensitive applications with IP whitelists

**Cost control:**
- Monitor API call volume and token consumption
- Set daily/monthly call limits
- Use cheaper models for internal tools

**Data privacy:**
- Check whether your Knowledge Base contains sensitive information
- Don't connect public-facing applications to internal knowledge bases
- For highly sensitive data, consider private deployment (covered in the next chapter)

The security specialist nodded after hearing all this: *"Sounds like you've thought it through pretty thoroughly."*

Lynn thought to herself: some of those lessons were learned the hard way.
